# 无线采集系统代码运行顺序讲解

这份文档按单片机上电后的真实执行顺序讲解当前分层后的固件。现在 `src/*/main.cpp` 只负责板级配置和启动，主要业务逻辑已经移动到 `lib/` 下的节点级 App 和驱动库中。

## 1. 上电后的总流程

1. ESP32-S3 复位，进入 Arduino 运行时。
2. Arduino 调用当前固件目录中的 `setup()`。
3. `setup()` 调用对应 App 的 `begin()`：
   - Master 调用 `MasterApp::begin()`。
   - Slave 调用 `ImuSlaveApp::begin()`。
   - Motor 调用 `MotorApp::begin()`。
4. App 完成硬件初始化、队列/信号量创建、ESP-NOW/BLE 初始化和 FreeRTOS 任务创建。
5. Arduino 进入 `loop()`：
   - Master 和 Slave 的 `loop()` 直接挂起，实际工作都在任务和回调中。
   - Motor 的 `loop()` 调用 `MotorApp::loopOnce()`，处理 BLE 命令队列和电机状态机。

## 2. 代码分层后的阅读入口

| 层级 | 位置 | 作用 |
| --- | --- | --- |
| 板级入口 | `src/master`、`src/slave-01`、`src/slave-02`、`src/motor` | 配置引脚、source、slot、BLE UUID 等板级参数，并启动 App。 |
| 节点应用层 | `lib/master_node`、`lib/imu_slave_node`、`lib/motor_node` | 管理任务、队列、状态机、无线/BLE 回调和业务流程。 |
| 驱动层 | `lib/ads1298_driver`、`lib/mpu6500_driver`、`lib/bts7960_motor` | 只负责硬件寄存器、SPI、PWM 等底层操作，不创建任务。 |
| 协议层 | `lib/capture_common` | 定义 ESP-NOW、ACK、USB CDC 包格式、CRC 和大小端编解码。 |

阅读源码建议顺序：

1. 先看对应 `src/*/main.cpp`，确认这块板是什么角色。
2. 再看对应 App：`MasterApp`、`ImuSlaveApp` 或 `MotorApp`。
3. 最后看被 App 调用的驱动和协议库。

## 3. Master 运行顺序

入口文件是 `src/master/main.cpp`。它配置 ADS1298 引脚，然后构造 `MasterApp`。

`MasterApp::begin()` 的执行顺序：

1. 初始化 USB CDC 串口，波特率保持 `2000000`。
2. 调用 `Ads1298Driver::begin()` 初始化 SPI、复位 ADS1298、写寄存器、启动连续转换，并绑定 DRDY 中断。
3. 创建 `slaveRxQueue`，用于保存 ESP-NOW 收到的 Slave 原始 batch。
4. 创建 `serialQueue`，用于保存所有待发给 PC 的 `PcSample`。
5. 初始化 ESP-NOW：
   - Wi-Fi STA 模式；
   - 固定 `ESPNOW_CHANNEL = 1`；
   - 添加广播 peer；
   - 注册接收回调。
6. 创建 3 个任务：
   - `serialTxTask`：合批发送 USB CDC 数据。
   - `wirelessTask`：发 Beacon、收 IMU batch、回 ACK、去重、输出诊断帧。
   - `sensorTask`：响应 ADS1298 DRDY，读取 EMG。

Master 的中断和回调只做轻量工作：

- ADS1298 DRDY 中断只通知 `sensorTask`，不在中断里读 SPI。
- ESP-NOW 接收回调只把原始 batch 复制到 `slaveRxQueue`，不在回调里做解码和串口输出。

`sensorTask` 循环：

1. 等待 DRDY 中断通知。
2. 读取 `micros()` 作为 EMG 时间戳。
3. 调用 `Ads1298Driver::readChannels()` 读取 CH6/CH7/CH8。
4. 封装为 `PcSample`，写入 `serialQueue`。

`wirelessTask` 循环：

1. 每秒广播一次 `BeaconPacket`，为 Slave 提供 Master 时间基准。
2. 在 1 秒窗口内处理所有收到的 IMU batch。
3. 调用 `capture_common` 校验 CRC 并解码 batch。
4. 为每个 Slave 维护接收窗口：
   - `ackBaseSeq` 表示连续确认到的 batch；
   - `recvBitmap` 表示窗口内已收到但不连续的 batch；
   - 重复 batch 只 ACK，不重复转发给 PC。
5. 对有效新 batch 中的样本写入 `serialQueue`。
6. 每秒输出一条诊断帧，包含转发量、错误量、在线 Slave 数。

`serialTxTask` 循环：

1. 优先从 `serialFrameQueue_` 取状态事件、同步诊断和 USB 重发帧。
2. 如果没有控制/诊断帧，再从 `serialQueue` 取样本。
3. 最多合并 `PC_SERIAL_MAX_SAMPLES = 16` 条。
4. 调用 `capture::buildPcBatchPacket()` 生成 USB CDC 二进制帧，并缓存最近的 sample batch 供 PC 请求重发。
5. 通过 `Serial.write()` 完整写出；Master 只有这个任务直接写 USB CDC。

## 4. Slave 运行顺序

两个 Slave 使用同一个 `ImuSlaveApp`，区别只在 `src/slave-01/main.cpp` 和 `src/slave-02/main.cpp` 的配置：

| 固件 | source | 传感器模式 | 发送 slot |
| --- | --- | --- | --- |
| `slave-01` | `0x01` | Gyro | `+2 ms` |
| `slave-02` | `0x02` | Accel | `+10 ms` |

`ImuSlaveApp::begin()` 的执行顺序：

1. 设置 CPU 频率为 `80 MHz`。
2. 打开串口，波特率保持 `2000000`。
3. 调用 `Mpu6500Driver::begin()` 初始化 SPI 和 MPU6500；若板级配置提供 `intPin >= 0`，同时启用 DATA_RDY 中断。
4. 创建 ESP-NOW 发送完成信号量。
5. 初始化 ESP-NOW：
   - Wi-Fi STA 模式；
   - 固定 `ESPNOW_CHANNEL = 1`；
   - 设置 ESP-NOW 发射功率；
   - 注册发送完成回调和接收回调。
6. 创建 2 个任务：
   - `wirelessTask`：发送 `SYNC_PROBE`、同步诊断、错峰发送和重传 batch。
   - `sensorTask`：默认 1 kHz 软件采样；配置 DATA_RDY 后由中断时间戳采样；写 batch ring buffer。

Slave 接收回调处理四类包：

- 收到 `BeaconPacket`：保存 Master MAC，并在近期没有有效双向同步点时作为兜底同步点。
- 收到 `SYNC_REPLY`：记录 T4，按 NTP 公式计算 offset，把 `(local_mid, master_mid)` 写入拟合窗口。
- 收到 `CommandPacket`：切换 `SYNC / STREAM_PENDING / STREAM / IDLE` 状态。
- 收到 `ImuAckPacket`：释放已被 Master 确认的 batch ring slot。

`sensorTask` 循环：

1. 未进入 STREAM 时清空临时 batch；若启用 DATA_RDY，则顺手清掉积压通知。
2. 默认按 `SAMPLE_PERIOD_US = 1000` 控制 1 kHz 采样；若启用 DATA_RDY，则等待 MPU6500 中断通知并使用 ISR 第一时间记录的 `micros()`。
3. 使用最新动态平滑时钟模型换算到 Master 时间轴。
4. 对样本时间戳做单调保护，若模型微调导致倒退，则钳制为上一点 `+1 us`。
5. 根据节点配置读取陀螺仪或加速度。
6. 每 20 个样本组成一个 batch，写入 256 batch 的本地 ring buffer。
7. 如果 ring buffer 溢出，后续样本 `sample_seq` 会跳号，并在下一包 `rx_flags` 中标记链路故障。

`wirelessTask` 循环：

1. 等待已经有 Master MAC。
2. 在 `SYNC / STREAM_PENDING / STREAM` 中每 50 ms 发送一次 `SYNC_PROBE`。
3. 在同步阶段每 200 ms 发送同步诊断；STREAM 中静默更新拟合模型。
4. STREAM 后以 20 ms 为 superframe，根据节点配置进入自己的发送窗口。
5. 每个窗口最多发送 3 个 batch，优先发送最早未 ACK 的 batch。
5. 如果 batch 已发送过，会设置 `IMU_FLAG_RETRANSMIT` 并重新计算 CRC。
6. ACK 丢失时允许重复发送，由 Master 去重。

## 5. Motor 运行顺序

入口文件是 `src/motor/main.cpp`。它配置 BLE 名称、NUS UUID、BTS7960 引脚、PWM 参数和五档叩击参数。

`MotorApp::begin()` 的执行顺序：

1. 打开串口，波特率 `115200`。
2. 创建 BLE 命令队列。
3. 调用 `Bts7960Motor::begin()` 初始化 REN/LEN、RPWM/LPWM 和 LEDC PWM。
4. 初始化 BLE Nordic UART Service：
   - RX characteristic 接收上位机命令；
   - TX characteristic notify 返回 ACK/ERR。
5. 开始 BLE advertising，设备名保持 `Neuro_Hammer_BLE`。

`MotorApp::loopOnce()` 每次做两件事：

1. 从命令队列取出所有待处理命令：
   - `PING`
   - `STRIKE_1` 到 `STRIKE_5`
   - `MOTOR_FWD`
   - `MOTOR_REV`
   - `STOP`
2. 更新电机非阻塞状态机：
   - `StrikePull`：反向回撤；
   - `StrikeFire`：正向敲击；
   - `ManualForward` / `ManualReverse`：持续转动；
   - `STOP` 或 BLE 断开会立即停止。

## 6. 协议字段速查

ESP-NOW IMU batch：

```text
type | source | batchSeq | sampleStartSeq | count | flags | baseTimeUs | sample[20] | crc16
```

单个 IMU sample：

```text
dtUs:uint16 | x:int16 | y:int16 | z:int16
```

Master ACK：

```text
type=0x11 | source | ackBaseSeq | ackSampleSeq | recvBitmap | masterTimeUs | crc16
```

Master 到 PC 的 `PcSample`：

```text
source | timestamp_us | sample_seq | batch_seq | flags | x | y | z
```

CSV 中判断采集是否连续，优先看：

```matlab
all(diff(sample_seq) == 1)
```
