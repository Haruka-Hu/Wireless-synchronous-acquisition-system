# 历史调试复盘

这份笔记记录项目早期从“Slave 没信号、波形很脏、点数越差越多、parser reject 猛涨”一路调到稳定的过程。部分细节来自旧版 500Hz 双 IMU / MPU6050 验证阶段，当前代码已经演进为 ADS1298 + `slave-01` / `slave-02` + `motor` 的多固件框架。

保留这份文档的目的不是描述当前架构，而是沉淀排障方法：以后遇到类似的错帧、CRC、时间戳、串口读取或无线链路问题，可以按这里的思路定位。

## 1. 当时的稳定现象

当前稳定状态的典型指标：

```text
Master 5323 点 | Slave 5320 点
Slave rate 500/s | missing 0/s | link err 0/s
crc fail 0 | ts reject 0 | resync 6B
```

这说明几个链路都已经基本健康：

- Slave 采样和发送到 Master 是 500Hz。
- Master 没检测到 Slave 批量包丢失。
- Master 到 PC 的串口帧没有 CRC 错误。
- 上位机解析没有时间戳拒绝。
- Master/Slave 点数长期接近。

## 2. 调试时最容易误判的点

### 2.1 波形脏不一定是传感器脏

一开始 Slave 波形大量尖峰，很容易怀疑：

- MPU6050 供电不稳。
- ESP-NOW 发射干扰 I2C。
- Slave 板子坏了。
- ESP-NOW 丢包或误码。

但后来用 `slave_probe` 验证：Slave 单独采样并发射时，波形是干净的。这说明问题不在 MPU6050 传感器本身，也不是简单的无线发射干扰。

结论：看到脏波形，先分层验证，不要一上来就换硬件。

### 2.2 点数少不一定是无线丢包

曾经出现 Master 点数明显多于 Slave 点数。直觉会怀疑 ESP-NOW 丢包，但 Master 诊断显示：

```text
Slave rate 500/s
missing 0/s
link err 0/s
```

这说明 Slave 到 Master 的链路其实是好的，真正少的是 Master 到 PC 或 PC 解析阶段。

结论：必须区分“固件内部收到多少”和“上位机解析出多少”。

## 3. 关键问题与修复

### 3.1 C++ 高低字节拼接隐患

风险代码：

```cpp
int16_t ax = Wire.read() << 8 | Wire.read();
```

这个写法不利于调试，也可能受到表达式求值顺序和类型提升影响。最终改成：

```cpp
const uint8_t axHigh = static_cast<uint8_t>(Wire.read());
const uint8_t axLow = static_cast<uint8_t>(Wire.read());
ax = static_cast<int16_t>((static_cast<uint16_t>(axHigh) << 8) | axLow);
```

经验：

- I2C 多字节寄存器读取，尽量一字节一行。
- 高低位拼接前先转成无符号宽类型。
- 不要为了短一行代码牺牲确定性。

### 3.2 I2C 读取必须检查 `endTransmission(false)`

错误倾向是写完寄存器地址后直接 `requestFrom()`。最终做法：

```cpp
const uint8_t txStatus = Wire.endTransmission(false);
if (txStatus != 0) {
  return false;
}
```

只有寄存器地址阶段成功，才继续读 6 字节。否则 I2C 设备没 ACK、总线异常、模块没供电时都会制造假数据。

经验：

- `endTransmission()` 返回值是 I2C 健康状态的一手证据。
- 嵌入式里不要忽略错误码。

### 3.3 Broadcast ESP-NOW 不适合数据回传

早期 Slave 用广播发 IMU 数据。问题是广播没有硬件 ACK，也没有重传确认。最终改成：

- Master 继续广播 Beacon。
- Slave 从 Beacon 回调里学习 Master MAC。
- Slave 后续用单播发 IMU 批量包。
- 发送后等待 send callback，不成功最多重试 3 次。

经验：

- 广播适合发现和同步，不适合持续数据上传。
- 数据通道优先用单播。
- 要看发送回调，而不是只看 `esp_now_send()` 的返回值。

### 3.4 多任务同时写串口会破坏字节流

早期 Master 有两个路径可能写串口：

- Core 1 本地采样任务写 Master 样本。
- Core 0 无线接收路径写 Slave 样本。

如果两个上下文同时 `Serial.write()`，字节可能交错。最终改成：

- 所有数据生产者只调用 `writeSerialSample()`。
- `writeSerialSample()` 只投递 `g_serialQueue`。
- 只有 `serialTxTask` 一个任务真正 `Serial.write()`。

经验：

- 串口是字节流，不是消息队列。
- 多个任务写同一个流式外设时，必须有单一 owner。
- Queue 是解耦多生产者和单消费者的标准方式。

### 3.5 13 字节裸帧会被 payload 假包头击穿

最初 PC 协议类似：

```text
AA 55 | source | timestamp | x | y | z
```

问题是 payload 中可能自然出现 `AA 55`。一旦 Python 解析错位，它可能把 payload 里的 `AA55` 当作新包头，后续数值就会变成巨大尖峰。

修复过程：

1. 先给单帧追加 CRC，能识别假帧。
2. 再升级成批量帧，减少包头数量和 Python 解析压力。

最终 PC 帧：

```text
AA 55 | type | count | sequence | sample... | crc16
```

经验：

- 帧头只能帮助“寻找可能的边界”。
- CRC 才能证明“这个边界是真的”。
- 高速二进制串口不要用太短、无校验的小包。

### 3.6 Python 的 `in_waiting` 在 macOS USB Serial/JTAG 上不可靠

现象：

```text
GUI 打开了，但没数据。
```

原因之一是 macOS 下 ESP32-S3 USB Serial/JTAG 的 `in_waiting` 有时长期返回 0，但阻塞式 `read(timeout)` 仍然能读到数据。最终上位机改成：

```python
chunk = self.serial_port.read(4096)
```

经验：

- 串口 API 在不同 USB CDC 实现上有细节差异。
- 对连续流，高吞吐读取可以直接用阻塞 read + timeout。

### 3.7 用样本序号当横轴会放大点数差异

早期显示如果用“第 N 个点 / 500Hz”作为横轴，一旦某一路解析少了点，就会显示成两路越来越偏。最终改为用固件 `timestamp_us` 作为横轴。

经验：

- 同步采集系统的横轴应该来自采样时刻，而不是 PC 接收顺序。
- PC 接收时间只能用于吞吐调试，不适合做物理同步分析。

### 3.8 时间 offset 不能每秒硬切

Slave 每秒收到 Beacon，如果直接把 offset 硬更新到新值，Beacon 空口延迟抖动会让时间戳每秒跳一下。最终策略：

- offset 小变化低通平滑。
- offset 大变化立即重锁。
- 采样时间戳保持单调，每点最多修正 200us。

经验：

- 时间同步既要跟随真实漂移，也要拒绝测量抖动。
- 复位/重烧录属于大事件，必须和普通抖动区分处理。

## 4. 指标含义与定位方法

### 4.1 `Slave rate`

来源：Master 诊断帧。

含义：Master 每秒成功解码并转发的 Slave 样本数。

判断：

- 接近 500/s：Slave 到 Master 基本正常。
- 明显低于 500/s：看 `missing` 和 `link err`。

### 4.2 `missing`

来源：Master 根据 Slave 批量包 sequence 推算。

含义：Slave 到 Master 途中丢了多少样本。

判断：

- `missing 0/s`：无线批量包没有明显丢失。
- 持续非 0：检查 ESP-NOW 单播、距离、信道、重试、供电。

### 4.3 `link err`

来源：Master 诊断帧。

含义：Master 端一秒内检测到的链路/处理错误增量，包含 CRC、解码、队列、串口短写等。

判断：

- `link err 0/s`：固件内部链路稳定。
- 增长：需要看代码里具体错误计数，判断是无线 CRC、队列满还是串口堵。

### 4.4 `crc fail`

来源：Python 上位机。

含义：PC 收到的串口帧 CRC 不匹配。

常见原因：

- Master 到 PC 字节流错位。
- 上位机中途从流中间开始读。
- 固件和上位机协议版本不一致。
- 串口数据被调试 `Serial.print()` 混入。

稳定运行时应为 0 或偶尔极少量。

### 4.5 `ts reject`

来源：Python 上位机。

含义：CRC 通过后，样本时间戳仍然不符合预期，例如同源相邻时间间隔太小或太大。

常见原因：

- 协议版本不一致导致解析字段错位。
- 时间同步重锁时出现大跳变。
- 旧数据残留。

当前稳定状态应为 0。

### 4.6 `resync`

来源：Python 上位机。

含义：为了重新找到 `AA55` 帧头，解析器丢弃了多少字节。

判断：

- 启动时有几个字节很正常，因为 Python 可能从字节流中间开始读。
- 运行中持续猛涨，说明串口流里有非协议字节或解析不断错位。

## 5. 推荐排查顺序

以后如果系统又不稳，建议按这个顺序查：

1. 看 Master 串口上位机状态：`Slave rate/missing/link err/crc fail/ts reject/resync`。
2. 如果 `Slave rate` 是 500 且 `missing` 是 0，但 Slave 点数少，优先查 Master 到 PC 协议和 Python 解析。
3. 如果 `missing` 增长，优先查 ESP-NOW 单播、距离、信道、供电。
4. 如果 `crc fail` 增长，确认固件和 Python 协议版本一致，且固件没有 `Serial.print()` 混入二进制流。
5. 如果单路波形脏，刷 `slave_probe` 或写单板测试固件，先证明传感器本身是否干净。
6. 如果两路时间错位，保存 CSV，用 `tools/analyze_imu_delay.py` 估计延迟，不要只靠肉眼。

## 6. 这次项目沉淀出的工程规则

- 二进制协议上线前就加 CRC，不要等出问题后再补。
- 回调函数只搬运数据，不做业务处理。
- 实时采样任务不要直接碰无线和串口。
- 所有跨核心共享状态都要有同步机制。
- 上位机显示要用设备时间戳，不要用接收顺序假装时间。
- 每个链路都要有诊断指标，不然只能盲猜。
- 验证假设时尽量做最小实验，例如 `slave_probe`，不要同时改很多变量。

## 7. 我们最终为什么能稳定

稳定不是某一个 magic fix 的结果，而是每层都加了边界：

- I2C 层：检查 ACK、逐字节拼接、异常点重读。
- Slave 采样层：500Hz 固定周期、时间戳单调、未同步不采样。
- ESP-NOW 层：Beacon 广播发现、数据单播、send callback、重试、sequence、CRC。
- Master 转发层：ESP-NOW 回调只入队，串口单任务 owner。
- PC 协议层：批量帧、长度、类型、序号、CRC、自恢复解析。
- GUI 层：独立串口线程、按时间戳画图、状态栏显示关键指标。

这套分层思想以后可以复用到很多嵌入式项目里：先把实时性、通信、协议、显示拆开，再逐层验证。
